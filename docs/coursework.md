## Введение

**Актуальность темы исследования** обусловлена необходимостью создания надёжных программных решений для автоматизации процесса сбора, хранения и отображения экономических событий — макроэкономических релизов, решений центральных банков, индексов деловой активности и прочих публикаций, которые оказывают прямое влияние на финансовые рынки. Современный участник рынка (трейдер, аналитик, риск-менеджер) ежедневно оперирует десятками подобных индикаторов, и скорость получения обновлённых данных, а также удобство их фильтрации становятся одним из ключевых факторов эффективности принимаемых решений. Ручной сбор данных с публичных календарей, сайтов регуляторов и брокерских площадок крайне трудоёмок, а результаты такого сбора быстро теряют релевантность. Внедрение автоматизированного модуля позволит повысить эффективность в области финансово-экономического анализа и сократить временные и трудозатраты, связанные с подготовкой информационной базы для последующего анализа и торговых решений.

Дополнительным фактором актуальности является тот факт, что большинство публичных экономических календарей (Investing.com, ForexFactory, MyFXBook и др.) предоставляют данные лишь в виде веб-страниц без открытого API, при этом локализованные русскоязычные источники (в частности, календарь брокера «Альфа-Форекс») содержат подробные описания индикаторов, но не имеют документированной интеграции. Поэтому разработка собственного интеграционного модуля, инкапсулирующего работу с разнородными источниками данных, является практически значимой задачей.

**Объектом исследования** является процесс управления данными экономических событий в информационно-аналитических системах финансового профиля, а **предметом исследования** — разработка программного модуля для автоматизированной обработки, агрегации и визуализации экономических событий с возможностью многоязычного представления и интерактивной фильтрации.

**Целью курсовой работы** является создание программного модуля для автоматизации процесса агрегации и отображения экономических событий из внешних источников данных в едином веб-интерфейсе. Для достижения цели необходимо выполнить следующие задачи:

1. Исследовать предметную область экономических календарей, классифицировать источники данных и типичные сценарии их использования.
2. Проанализировать существующие методы и решения задачи сбора и отображения экономических событий; выбрать и обосновать архитектурный и технологический подход.
3. Разработать модели предметной области и спроектировать архитектуру модуля в нотациях UML (Use Case), IDEF0, DFD, функциональных схем.
4. Сформулировать функциональные и нефункциональные требования к программному модулю.
5. Реализовать серверную часть (REST API на FastAPI + SQLAlchemy) и клиентскую часть (SPA на React + Vite).
6. Реализовать интеграцию с внешними источниками (AlfaForex, FRED) и механизм кэширующей синхронизации.
7. Провести тестирование разработанного модуля и оценить соответствие реализации требованиям.

**Практическая значимость** курсовой работы заключается в создании модуля, который может быть использован для автоматизации процессов мониторинга экономических событий в брокерских, аналитических и исследовательских подразделениях финансовых организаций, а также для личного использования частными трейдерами. Это позволит повысить точность получаемой информации и сократить время выполнения задач по подготовке новостного фона и обзоров. Модуль спроектирован так, что его можно развернуть локально, встраивать в корпоративные порталы или публиковать в виде статического фронтенда (GitHub Pages) с отдельно размещённым REST-сервисом.

**Теоретико-методологическую базу** работы составили исследования и разработки в области структурного и объектно-ориентированного программирования, проектирования распределённых систем, веб-архитектуры REST, моделирования данных в реляционных СУБД, а также методологические подходы, изложенные в работах Р. **Методы исследования:** системный анализ предметной области, сравнительный анализ существующих решений, структурное и объектно-ориентированное моделирование, проектирование архитектуры с применением нотаций UML, IDEF0 и DFD, итеративная разработка программного обеспечения, функциональное и интеграционное тестирование, эмпирический анализ поведения системы при работе с реальными источниками данных.

**Структура работы.** Работа состоит из введения, трёх глав, заключения, списка использованных источников и приложений. В первой главе изложены теоретические основы и анализ проблемы: описана предметная область, выполнен обзор существующих решений и обоснован выбор подхода. ## Глава 1. Теоретические основы и анализ проблемы

### 1.1. Описание предметной области и постановка проблемы

Экономический календарь — специализированный информационный ресурс, содержащий расписание публикаций макроэкономических показателей, заседаний центральных банков, выступлений официальных лиц и других событий, способных оказать влияние на финансовые рынки. Каждая запись в подобном календаре представляет собой кортеж атрибутов: дата и время публикации (часто с точностью до минуты и привязкой к часовому поясу), страна/регион, регулятор или статистическое ведомство, наименование показателя, валюта, мера «важности» (volatility/importance), а также три ключевых значения: фактическое (actual), прогнозное (consensus/forecast) и предыдущее (previous). Для большинства индикаторов также характерно наличие текстового описания, раскрывающего методологию расчёта и влияние на рынок.

Пользователями таких систем являются:

- частные и институциональные трейдеры, принимающие краткосрочные торговые решения на волатильности, сопровождающей публикации;
- финансовые аналитики и экономисты, использующие исторические и ожидаемые значения в исследовательских моделях;
- портфельные и риск-менеджеры, планирующие ребалансировку и стресс-тестирование с учётом календаря «риск-событий»;
- корпоративные казначеи, отвечающие за управление валютными и процентными рисками;
- студенты и преподаватели экономических и финансовых дисциплин, использующие данные в учебных целях.

**Постановка проблемы.** В большинстве случаев указанные пользователи сталкиваются со следующими трудностями:

1. **Разнородность источников.** Данные распределены между сайтами центральных банков, статистических агентств (U.S. Bureau of Economic Analysis, Bureau of Labor Statistics, Eurostat, Росстат, National Bureau of Statistics of China и др.), агрегаторами (Investing.com, ForexFactory, TradingEconomics) и брокерскими календарями (Alfa-Forex, RoboForex, Dukascopy). Каждый источник использует собственные форматы, часовые пояса и наборы индикаторов.
2. **Отсутствие стабильного публичного API.** Большинство популярных календарей не публикуют REST API, ориентируясь на HTML-страницы с динамической подгрузкой через AJAX. Это приводит к необходимости разработки клиентов-парсеров, чувствительных к изменениям вёрстки.
3. **Актуальность и кэширование.** Данные могут меняться несколько раз в сутки (публикация фактического значения, корректировка предыдущих значений, переносы релизов), что требует регулярной пересинхронизации при необходимости избегать избыточных запросов к внешним системам.
4. **Локализация и описания.** Русскоязычные источники (например, Альфа-Форекс) содержат подробные описания индикаторов, однако при переключении на английский, китайский или испанский языки существующие открытые решения либо теряют тексты, либо возвращают исходные русские описания.
5. **Удобство фильтрации и представления.** Пользователю необходимо интерактивно выбирать страну, важность, временной диапазон (сегодня, завтра, неделя, произвольная дата) и быстро видеть ключевые метрики (actual/forecast/previous) в компактном виде.

Таким образом, ставится задача **разработки унифицированного программного модуля**, который скрывает разнородность внешних источников, обеспечивает постоянную актуальность данных в локальной БД, предоставляет REST API и многоязычный веб-интерфейс с гибкой фильтрацией и интерактивным просмотром описаний событий.

### 1.2. Обзор существующих методов и решений

**Готовые веб-сервисы.** Investing.com, ForexFactory, TradingEconomics, MyFXBook предоставляют веб-интерфейсы календарей с широкой поддержкой стран и индикаторов. Их недостатки с точки зрения встраивания в собственные системы: отсутствие открытого API (либо API с платной подпиской от $100–$500/мес для TradingEconomics), ограничения на автоматизированный парсинг (reCAPTCHA, Cloudflare, rate-limit), неполный перевод описаний и невозможность расширения схемы данных собственными индикаторами.

**Публичные API макроданных.** Federal Reserve Economic Data (FRED) публикует REST API по рядам макроэкономических показателей США и отдельных стран, включая даты публикаций релизов. API стабилен, однако предоставляет не «события», а «ряды», и требует собственной логики формирования календарных записей. Eurostat и Росстат публикуют данные в форматах SDMX/JSON, но без «календарного» представления. Публичные API удобно использовать как дополнительный источник, но недостаточно в качестве единственного.

**Локальные настольные решения.** Встречаются плагины к торговым терминалам MetaTrader (MT4/MT5 — Economic Calendar), которые привязаны к конкретной платформе, не имеют веб-интерфейса и не масштабируются до корпоративного использования.

**Подходы к парсингу HTML.** Рассматривались три варианта:

- статический парсинг через `requests` + `BeautifulSoup` — быстро, но не справляется со страницами, данные которых подгружаются JavaScript;
- прямое обращение к внутренним XHR-эндпоинтам сайта — быстро и надёжно, если такой эндпоинт удаётся обнаружить;
- безголовый браузер (`Playwright`, `Selenium`, `Puppeteer`) — медленно (секунды на запрос), но работает с любыми SPA.

Оптимальной стратегией признан **гибридный подход**: в первую очередь используется внутренний JSON-эндпоинт источника (у Альфа-Форекс это `/api/economic-calendar/?action=events&culture=...&timeZone=...`), а в случае его недоступности выполняется fallback на рендеринг страницы в Playwright с последующим извлечением структурированных строк таблицы. Такой подход сочетает скорость и устойчивость.

**Архитектурные шаблоны.** Для модуля рассматривались: монолитное веб-приложение на Django/Flask с server-side rendering; микросервисная архитектура с отдельными сервисами для каждого источника; SPA-фронтенд + REST-бэкенд. Исходя из компактности задачи и требования к отзывчивости интерфейса, выбран третий вариант: единый REST-бэкенд с несколькими модулями-адаптерами (AlfaForex, FRED, manual), SPA-фронтенд с клиентской маршрутизацией и интернационализацией.

**Хранение данных.** Сравнивались SQLite, PostgreSQL и MongoDB. Для учебно-прикладной задачи с ограниченным объёмом данных (тысячи–десятки тысяч записей) и возможностью распространения проекта в виде одного каталога выбрана SQLite; при этом используется ORM SQLAlchemy, что обеспечивает лёгкий переход к PostgreSQL без изменения прикладного кода.

**Организация синхронизации.** Возможные варианты: периодический запуск по расписанию (cron/Celery beat), ленивая синхронизация при обращении пользователя, событийная синхронизация по вебхукам. Выбрана комбинация первого и второго вариантов: при запуске приложения выполняется принудительное обновление, при каждом запросе к API проверяется TTL (300 секунд по умолчанию) и при его истечении запускается пересинхронизация. Это исключает необходимость настройки внешнего планировщика и гарантирует актуальность данных для активных пользователей.

### 1.3. Выбор подхода и его обоснование

На основании проведённого анализа принят следующий подход к проектированию модуля:

1. **Клиент-серверная архитектура** с REST API и SPA-фронтендом.
2. **Слой адаптеров** для внешних источников данных с унифицированным внутренним интерфейсом «upsert внешнего события».
3. **Реляционная СУБД SQLite** под управлением SQLAlchemy (ORM) с поддержкой лёгкой миграции структуры таблицы `events` для обратной совместимости.
4. **Гибридная стратегия получения данных** (внутренний JSON API сайта + Playwright как fallback).
5. **Кэширующая синхронизация по TTL** и ленивое обновление при запросах пользователя.
6. **Многоязычный интерфейс** (ru/en/es/zh) с динамической загрузкой описаний события на языке пользователя через отдельный REST-эндпоинт.

### 1.4. Выводы по главе

В первой главе описана предметная область экономических календарей, выделены группы пользователей и сформулирована постановка проблемы: необходимо разработать унифицированный программный модуль, решающий задачи агрегации разнородных источников, локального хранения, регулярной актуализации и многоязычного удобного отображения экономических событий. Выполнен обзор существующих решений: готовые веб-сервисы, публичные API макростатистики, настольные терминальные плагины и различные подходы к парсингу данных. Показано, что ни одно из существующих решений по отдельности не удовлетворяет всем сформулированным требованиям, что обосновывает актуальность собственной разработки. В качестве архитектурного подхода выбрана клиент-серверная SPA-архитектура с REST API, слоем адаптеров источников, реляционным хранилищем и гибридной стратегией загрузки данных. Этот подход положен в основу предпроектной части (гл. 2) и реализации (гл. 3).

## Глава 2. Предпроектная часть

### 2.1. Диаграммы и моделирование

#### 2.1.1. Диаграмма вариантов использования (Use Case)

Для описания сценариев взаимодействия с модулем выделены два актёра: **Пользователь** (трейдер/аналитик) и внешний актёр **Внешний источник данных** (Альфа-Форекс, FRED). Диаграмма вариантов использования представлена на рис. 2.1 (текстовое представление в нотации PlantUML приведено ниже).

```
@startuml UseCase
left to right direction
actor "Пользователь" as User
actor "Внешний источник\n(AlfaForex, FRED)" as Ext

rectangle "Модуль экономических событий" {
  usecase "UC1. Просмотр списка событий" as UC1
  usecase "UC2. Фильтрация по стране,\nважности и дате"      as UC2
  usecase "UC3. Просмотр описания события\nна выбранном языке" as UC3
  usecase "UC4. Просмотр календарного\nпредставления"         as UC4
  usecase "UC5. Переключение языка\nинтерфейса"              as UC5
  usecase "UC6. Переключение темы\n(светлая/тёмная)"         as UC6
  usecase "UC7. Добавление\nсобственного события" as UC7
  usecase "UC8. Принудительное\nобновление данных" as UC8
  usecase "UC9. Синхронизация с внешним\nисточником" as UC9
}

User --> UC1
User --> UC2
User --> UC3
User --> UC4
User --> UC5
User --> UC6
User --> UC7
User --> UC8
UC1 ..> UC9 : <<include>>\n(ленивая)
UC8 ..> UC9 : <<include>>
UC9 --> Ext
@enduml
```

Кратко описание ключевых вариантов использования:

- **UC1. Просмотр списка событий** — пользователь открывает страницу «События» и получает отсортированный список всех событий из БД.
- **UC2. Фильтрация** — пользователь применяет фильтры по стране, важности и дате (сегодня/завтра/неделя/произвольная дата).
- **UC3. Просмотр описания** — пользователь выбирает строку, вызывается модальное окно с описанием события на текущем языке.
- **UC4. Календарное представление** — группировка событий по датам с интерактивным выбором.
- **UC7. Добавление события** — пользователь (или администратор) вручную создаёт событие через POST `/events`.
- **UC8. Принудительное обновление** — запуск синхронизации по запросу (POST `/events/refresh`).
- **UC9. Синхронизация** — внутренний сценарий, в котором модуль обращается к внешним источникам и выполняет upsert записей в БД.

#### 2.1.2. Функциональная модель (IDEF0)

IDEF0-модель описывает систему как иерархию функциональных блоков. На контекстной диаграмме A-0 представлен единственный блок «Обрабатывать экономические события», имеющий вход (сырые данные из внешних источников, команды пользователя), выход (структурированные данные событий, HTTP-ответы API, визуализация в UI), управление (регламент TTL, требования к данным, настройки культуры/часового пояса) и механизмы (FastAPI, SQLAlchemy, SQLite, Playwright, React). На диаграмме A0 блок декомпозирован на пять функций:

- **A1. Получение данных из внешних источников.** Входы: запрос на синхронизацию, расписание TTL. Выходы: «сырой» JSON/HTML ответ. Управление: конфигурация источников (URL, культура, таймзона). Механизмы: `urllib`, `httpx`, `Playwright`.
- **A2. Нормализация и преобразование данных.** Входы: сырой ответ. Выходы: унифицированная модель события (`EventCreate`/`EventRead`). Управление: правила парсинга (регулярные выражения для дат, словарь русских месяцев, маппинг волатильности → важности). Механизмы: Python + регулярные выражения.
- **A3. Хранение и актуализация в БД.** Входы: нормализованная модель. Выход: запись в таблице `events`. Управление: ограничения уникальности (`external_id`), TTL. Механизмы: SQLAlchemy ORM, SQLite.
- **A4. Предоставление REST API.** Входы: HTTP-запросы клиента. Выходы: JSON-ответы. Управление: OpenAPI-схема, CORS-политика. Механизмы: FastAPI, Pydantic.
- **A5. Визуализация и взаимодействие с пользователем.** Входы: JSON-ответы API, действия пользователя. Выходы: отрисованные страницы и модальные окна. Управление: локализация, тема. Механизмы: React, Vite, React Router.

Текстовое представление диаграммы A0:

```
              ┌──────────────────────────┐  Конфигурация (TTL, culture, TZ)
              ▼                          │
  Внешние ──► A1 Получение ─────┐        │
  источники                     ▼        │
                         A2 Нормализация ┤ Правила парсинга
                                ▼        │
                         A3 Хранение ────┤ Ограничения уникальности
                                ▼        │
                         A4 REST API ────┤ OpenAPI, CORS
                                ▼        │
          Пользователь ◄── A5 UI ────────┘ Локализация, темы
```

Каждый блок при необходимости может быть декомпозирован дальше (например, A1 разбивается на «A1.1 Запрос к JSON-эндпоинту», «A1.2 Fallback на Playwright», «A1.3 Агрегация описаний по культурам»).

#### 2.1.3. Диаграмма потоков данных (DFD)

Диаграмма потоков данных отражает движение данных внутри модуля и между ним и внешним окружением. В нотации Гэйна–Сарсона выделены:

**Внешние сущности:** Пользователь (U), Внешний источник (E).

**Хранилища данных:** D1 — `events` (таблица SQLite), D2 — кэш описаний (in-memory `_desc_cache`), D3 — метка последней синхронизации (`_last_sync_epoch`).

**Процессы:**

- P1. Приём HTTP-запроса (FastAPI роутер).
- P2. Проверка TTL и решение о синхронизации.
- P3. Загрузка данных из внешнего источника.
- P4. Парсинг и нормализация.
- P5. Upsert в БД.
- P6. Выборка событий по фильтрам.
- P7. Загрузка описания события по языку.
- P8. Сериализация ответа (Pydantic).
- P9. Отрисовка UI (клиентский процесс).

**Основные потоки данных:**

```
U  ──(HTTP GET /events?country=...)──► P1 ──► P2
P2 ──(TTL истёк)──► P3 ──► E ──(JSON/HTML)──► P3 ──► P4 ──► P5 ──► D1
P2 ──(TTL не истёк)──► P6 ◄──► D1
P6 ──(list[Event])──► P8 ──(JSON)──► U ──► P9 (рендер таблицы)

U  ──(HTTP GET /events/{id}/description?lang=en)──► P1 ──► P7
P7 ◄──► D2 (кэш); при промахе P7 ──► E ──► P7 ──► D2
P7 ──► P8 ──► U

U  ──(HTTP POST /events/refresh)──► P1 ──► P3 (force=True) ──► P4 ──► P5 ──► D1
P2 ◄──► D3 (чтение/запись timestamp)
```

Диаграмма подчёркивает, что ни один процесс не обращается к внешнему источнику напрямую, минуя P3, и что все модификации D1 проходят через процесс P5, инкапсулирующий операцию upsert.

## Глава 3. Реализация системы

### 3.1. Архитектура и логика работы модуля

Приложение реализовано по клиент-серверной схеме и состоит из трёх слоёв: слоя данных (SQLAlchemy ORM + SQLite), слоя бизнес-логики и интеграций (Python-модули `crud`, `alfaforex_sync`, `fred_sync`, `seed`) и слоя представления (FastAPI-роутер + React SPA). Структура исходного кода представлена ниже.

```
│   ├── models.py            # ORM-модель Event
│   ├── schemas.py           # Pydantic-схемы EventCreate/EventRead
│   ├── crud.py              # CRUD-операции и upsert
│   ├── alfaforex_sync.py    # адаптер AlfaForex (JSON + Playwright)
│   ├── fred_sync.py         # адаптер FRED (необязательный)
│   └── seed.py              # начальное наполнение

├── index.html
├── package.json
├── vite.config.js
└── src/
    ├── main.jsx
```

**Модель данных.** В таблице `events` хранятся все события вне зависимости от источника. Ключевые поля: `id`, `title`, `date`, `country`, `regulator`, `importance` (`low|medium|high`), `event_time` (HH:MM), `currency`, `actual`, `forecast`, `previous`, `description`, `source` (`alfaforex|fred|manual`), `external_id` (уникальный в пределах `NOT NULL`), `created_at`. Уникальный частичный индекс `ix_events_external_id` гарантирует отсутствие дубликатов при повторных синхронизациях.

**Жизненный цикл запроса.** При старте приложения (`startup` хук) создаются таблицы, выполняется миграция колонок, начальный seed и принудительная синхронизация. При каждом запросе `GET /events` вызывается `refresh_if_stale`, который проверяет глобальный timestamp `_last_sync_epoch` и, если TTL истёк, запускает `sync_alfaforex_events`. Последний обращается к JSON API источника (`_fetch_events_payload`), а при неудаче — к Playwright (`_fetch_rendered_rows`). Полученные записи приводятся к внутренней модели и передаются в `upsert_external_event`, которая вставляет новые или обновляет изменившиеся записи.

**Описание на другом языке.** Эндпоинт `/events/{id}/description?lang=en` отдельно обращается к AlfaForex с параметром `culture=en-US`, получает `HTMLDescription`, очищает HTML-разметку и кэширует результат в памяти на 10 минут по ключу культуры. Это экономит трафик при многократном открытии одного и того же события.

### 3.2. Программная реализация

#### 3.2.1. Модель SQLAlchemy

Модель описывает все поля, необходимые для представления события, включая расширения для метрик и источника.

```python
# backend/app/models.py
class Event(Base):
    __tablename__ = "events"

    id: Mapped[int] = mapped_column(Integer, primary_key=True, index=True)
    title: Mapped[str] = mapped_column(String(255), nullable=False)
    date: Mapped[Date] = mapped_column(Date, nullable=False, index=True)
    country: Mapped[str] = mapped_column(String(100), nullable=False, index=True)
    regulator: Mapped[str] = mapped_column(String(150), nullable=False, index=True)
    importance: Mapped[str] = mapped_column(String(20), nullable=False, index=True)

    event_time: Mapped[Optional[str]] = mapped_column(String(16), nullable=True)
    remaining_time: Mapped[Optional[str]] = mapped_column(String(32), nullable=True)
    currency: Mapped[Optional[str]] = mapped_column(String(16), nullable=True)
    actual: Mapped[Optional[str]] = mapped_column(String(64), nullable=True)
    forecast: Mapped[Optional[str]] = mapped_column(String(64), nullable=True)
    previous: Mapped[Optional[str]] = mapped_column(String(64), nullable=True)
    description: Mapped[Optional[str]] = mapped_column(Text, nullable=True)
    source: Mapped[str] = mapped_column(
        String(32), nullable=False, server_default=text("'manual'"), index=True
    )
    external_id: Mapped[Optional[str]] = mapped_column(String(160), nullable=True)
```

#### 3.2.3. Идемпотентный upsert

Ядро интеграции со внешними источниками — функция `upsert_external_event`. Она выбирает запись по `external_id` и либо вставляет новую, либо обновляет поля, только если хотя бы одно из них изменилось, что минимизирует количество записываемых транзакций.

```python
# backend/app/crud.py (фрагмент)
def upsert_external_event(db: Session, *, external_id: str, title: str,
                          event_date, country: str, regulator: str,
                          importance: str, source: str, **rest) -> str:
    existing = db.scalar(
        select(models.Event).where(models.Event.external_id == external_id)
    )
    if existing is None:
        event = models.Event(
            title=title, date=event_date, country=country,
            regulator=regulator, importance=importance,
            source=source, external_id=external_id, **rest,
        )
        db.add(event)
        try:
            db.commit()
            return "inserted"
        except IntegrityError:
            db.rollback()
            return "skipped"

    changed = False
    for attr, value in [
        ("title", title), ("date", event_date), ("country", country),
        ("regulator", regulator), ("importance", importance), ("source", source),
        *rest.items(),
    ]:
        if getattr(existing, attr) != value:
            setattr(existing, attr, value); changed = True
    if not changed:
        return "skipped"
    db.add(existing); db.commit()
    return "updated"
```

#### 3.2.4. Гибридная стратегия получения данных

В `alfaforex_sync.py` реализованы две стратегии: быстрый путь (прямой JSON API) и fallback через Playwright.

```python
# backend/app/alfaforex_sync.py (фрагмент)
def _fetch_events_payload(*, culture: Optional[str] = None) -> list[dict[str, Any]]:
    endpoint = _alfaforex_api_endpoint(include_countrycode=False, culture=culture)
    with urllib.request.urlopen(endpoint, timeout=30) as resp:
        payload = json.loads(resp.read().decode("utf-8", "ignore"))
    return payload if isinstance(payload, list) else []


def sync_alfaforex_events(db: Session) -> dict[str, int]:
    inserted = updated = skipped = fetched = 0
    try:
        payload = _fetch_events_payload()
        fetched = len(payload)
        for item in payload:
            event_id = str(item.get("IdEcoCalendar") or "").strip()
            title = str(item.get("Name") or "").strip()[:255]
            if not event_id or not title:
                skipped += 1; continue
            day = date.fromisoformat(str(item["DateTime"]["Date"])[:10])
            ev_time = None if item.get("AllDay") else (
                f'{int(item["DateTime"]["Hour"]):02d}:'
                f'{int(item["DateTime"]["Minute"]):02d}'
            )
            status = crud.upsert_external_event(
                db,
                external_id=f"alfaforex:{event_id}",
                title=title, event_date=day, event_time=ev_time,
                country=str(item.get("Country") or "Не указано")[:100],
                regulator="Альфа-Форекс",
                importance=_importance_from_volatility(item.get("Volatility")),
                source="alfaforex",
                currency=item.get("Currency"),
                actual=item.get("DisplayActual"),
                forecast=item.get("DisplayConsensus"),
                previous=item.get("DisplayPrevious"),
                description=_clean_html_description(str(item.get("HTMLDescription") or "")),
            )
            inserted += status == "inserted"
            updated += status == "updated"
            skipped += status == "skipped"
    except Exception:
        # fallback: render the page with Playwright and parse visible rows
        rows = _fetch_rendered_rows()
        fetched = len(rows)
        # ... аналогичный цикл upsert, парсящий HTML-строки ...
    return {"fetched": fetched, "inserted": inserted, "updated": updated, "skipped": skipped}
```

Такая двойная стратегия устраняет проблему хрупкости, присущую парсингу HTML: если структура страницы изменится или JSON-эндпоинт станет недоступен, модуль продолжит работать на альтернативном канале.

#### 3.2.5. Кэширующая синхронизация

Для предотвращения избыточной нагрузки на внешний источник используется простой TTL-кэш:

```python
_last_sync_epoch: float = 0.0
ALFAFOREX_TTL_SECONDS = int(os.getenv("ALFAFOREX_SYNC_TTL_SECONDS") or "300")

def refresh_if_stale(db, force=False):
    global _last_sync_epoch
    now = time.time()
    if not force and _last_sync_epoch and now - _last_sync_epoch < ALFAFOREX_TTL_SECONDS:
        return None
    result = sync_alfaforex_events(db)
    _last_sync_epoch = now
    return result
```

Аналогичный (но пер-культурный) кэш используется для локализованных описаний в `get_description_by_external_id` со сроком жизни 10 минут.

#### 3.2.6. Миграция SQLite

Учебный проект должен корректно открываться на машине пользователя, даже если `events.db` создан ранней версией модуля. Функция `ensure_sqlite_columns` проверяет `PRAGMA table_info` и добавляет недостающие колонки:

```python
# backend/app/database.py (фрагмент)
def ensure_sqlite_columns() -> None:
    if engine.dialect.name != "sqlite":
        return
    with engine.begin() as conn:
        colnames = {r[1] for r in conn.execute(text("PRAGMA table_info('events')")).fetchall()}
        if not colnames:
            return
        missing = [
            ("event_time", "VARCHAR(16)"),
            ("source", "VARCHAR(32) NOT NULL DEFAULT 'manual'"),
            ("external_id", "VARCHAR(160)"),
            ("remaining_time", "VARCHAR(32)"),
            ("currency", "VARCHAR(16)"),
            ("actual", "VARCHAR(64)"),
            ("forecast", "VARCHAR(64)"),
            ("previous", "VARCHAR(64)"),
            ("description", "TEXT"),
        ]
        for name, ddl in missing:
            if name not in colnames:
                conn.execute(text(f"ALTER TABLE events ADD COLUMN {name} {ddl}"))
        conn.execute(text(
            "CREATE UNIQUE INDEX IF NOT EXISTS ix_events_external_id "
            "ON events (external_id) WHERE external_id IS NOT NULL"
        ))
```

```jsx
      if (isActive) setError(e.message);

      if (isActive) setLoading(false);

}, []);
```

#### 3.2.8. Тестирование

Тестирование модуля проводилось в три этапа.

1. **Ручное функциональное тестирование API.** Через `curl` проверены эндпоинты: `GET /health` (200 OK); `GET /events` без фильтров возвращает непустой список после синхронизации; `GET /events?country=Россия&importance=high` применяет фильтры; `POST /events/refresh` возвращает `{"fetched": N, "inserted": …, "updated": …, "skipped": …}`; `GET /events/{id}/description?lang=en` возвращает англоязычный текст при наличии `external_id`, русскоязычный fallback — при его отсутствии.
2. **Интеграционное тестирование адаптеров.** Ключевые проверки: корректность парсинга `DateTime` при `AllDay=True` (должно сохраняться только `date`); корректный маппинг `Volatility`→`importance`; устойчивость к частичному ответу API (пропуск записей с пустым `IdEcoCalendar` или `Name`); идемпотентность (повторный прогон `sync_alfaforex_events` на тех же данных не увеличивает количество записей и даёт `updated=0`).
3. **UI-тестирование.** В браузере проверены: отображение таблицы на русском/английском/испанском/китайском языках; корректная работа фильтра «сегодня»/«завтра»/«неделя»/«произвольная дата»; открытие модального окна описания и корректный перевод при смене языка; сохранение темы (light/dark) между перезагрузками; адаптивность при ширине экрана от 360px; сборка `vite build` без ошибок и работа на GitHub Pages в режиме `HashRouter`.

### Приложение Д. Инструкция по развёртыванию (для программиста)

```bash
# Backend
cd backend
python -m venv .venv && . .venv/bin/activate
pip install -r requirements.txt
python -m playwright install chromium
uvicorn app.main:app --reload --port 8000

# Frontend
cd ../frontend

# production build:
npm run build && npm run preview
```

Переменные окружения:

- `ALFAFOREX_PAGE_URL` — URL страницы календаря (по умолчанию `https://alfaforex.ru/economic-calendar/`);
- `ALFAFOREX_SYNC_TTL_SECONDS` — TTL автосинхронизации (сек, по умолчанию 300);
- `ALFAFOREX_CULTURE`, `ALFAFOREX_TIMEZONE` — культура и часовой пояс API источника;
- `FRED_API_KEY` — ключ FRED (опционально);
- `FRONTEND_ORIGINS`, `FRONTEND_ORIGIN_REGEX` — настройка CORS.

### Приложение Е. Инструкция пользователя

1. Перейдите по адресу веб-приложения (например, `http://localhost:5173`).
2. В верхней панели выберите язык интерфейса (русский / English / Español / 中文) и тему (светлая/тёмная).
3. На странице «События» примените необходимые фильтры по стране, важности и дате.
4. Для просмотра подробного описания события кликните по строке таблицы — откроется модальное окно. При смене языка описание автоматически перезагружается.
5. Для календарного представления откройте вкладку «Календарь»: события сгруппированы по датам, внутри дня отсортированы по времени.
6. Данные обновляются автоматически каждые 60 секунд; при необходимости можно выполнить принудительное обновление через `POST /events/refresh`.
