# КУРСОВАЯ РАБОТА

## Тема: «Разработка веб-приложения экономического макро-календаря с автоматической синхронизацией данных из внешних источников»

---

## ВВЕДЕНИЕ

Актуальность темы исследования обусловлена необходимостью создания надёжных программных решений для автоматизации процесса мониторинга макроэкономических событий. В условиях глобальной экономической интеграции участники финансовых рынков — трейдеры, аналитики, инвесторы — нуждаются в своевременном и структурированном доступе к информации о ключевых экономических публикациях: решениях по процентным ставкам, данных по ВВП, индексах потребительских цен, показателях рынка труда и других макроиндикаторах. Отсутствие единого централизованного инструмента с возможностью фильтрации, многоязычной поддержки и автоматического обновления данных приводит к дополнительным временным и трудозатратам при поиске и анализе информации. Разработка веб-приложения экономического календаря, агрегирующего данные из нескольких внешних источников и предоставляющего удобный интерфейс пользователя, позволит повысить эффективность работы специалистов финансовой сферы и сократить время на поиск релевантной информации.

Объектом исследования является процесс агрегации и представления макроэкономических событий в веб-среде. Предметом исследования выступает разработка программного модуля для автоматизированного сбора, хранения, обновления и отображения данных экономического календаря с использованием современных веб-технологий.

Целью курсовой работы является проектирование и реализация полнофункционального веб-приложения экономического макро-календаря, обеспечивающего автоматическую синхронизацию данных из внешних источников, многоязычную поддержку интерфейса и гибкую систему фильтрации событий. Для достижения поставленной цели необходимо выполнить следующие задачи:

1. Провести анализ предметной области и обзор существующих решений в сфере экономических календарей.
2. Определить функциональные и нефункциональные требования к разрабатываемому модулю.
3. Спроектировать архитектуру приложения, включая модели данных, API-интерфейсы и пользовательский интерфейс.
4. Реализовать серверную часть (backend) на основе фреймворка FastAPI с использованием SQLAlchemy и SQLite.
5. Реализовать клиентскую часть (frontend) на основе React с поддержкой интернационализации и адаптивного дизайна.
6. Реализовать модули синхронизации данных с внешними источниками (Alfa-Forex API, FRED API).
7. Протестировать работоспособность приложения и проанализировать результаты.

Практическая значимость курсовой работы заключается в создании модуля, который может быть использован для автоматизации процесса мониторинга экономических событий, что позволит повысить оперативность получения информации и сократить время анализа макроэкономической ситуации. Приложение может быть развёрнуто как на локальном сервере, так и на платформе GitHub Pages для широкого доступа.

Теоретико-методологическую базу работы составили исследования и разработки в области клиент-серверного программирования, проектирования REST API, моделирования реляционных баз данных, а также методологические подходы, изложенные в трудах Фаулера М. по архитектуре приложений предприятия, Ричардсона Л. и Руби С. по RESTful-сервисам, Бэнкса А. и Порселло Е. по React-разработке.

Методы исследования: анализ предметной области, сравнительный анализ существующих решений, объектно-ориентированное моделирование, функциональное моделирование (IDEF0, DFD, Use Case), прототипирование, тестирование программного обеспечения.

Структура работы включает введение, три главы, заключение, список использованных источников и приложения. В первой главе рассматриваются теоретические основы и проводится анализ предметной области. Вторая глава посвящена предпроектному анализу, моделированию и формированию требований. В третьей главе описываются реализация системы, архитектура, программный код и пользовательский интерфейс. В заключении подводятся итоги работы и определяются перспективы развития.

## ГЛАВА 1. ТЕОРЕТИЧЕСКИЕ ОСНОВЫ И АНАЛИЗ ПРОБЛЕМЫ

### 1.1. Описание предметной области и постановка проблемы

Экономический календарь представляет собой инструмент, систематизирующий информацию о запланированных макроэкономических публикациях и событиях. К таким событиям относятся: решения центральных банков по процентным ставкам (FOMC, ECB, BoE, BoJ), публикации индексов потребительских цен (CPI), данных по валовому внутреннему продукту (GDP), показателей рынка труда (Nonfarm Payrolls, уровень безработицы), индексов деловой активности (PMI), данных по розничным продажам и других индикаторов.

Для участников финансовых рынков экономический календарь является критически важным инструментом принятия решений. Публикация макроэкономических данных оказывает непосредственное влияние на котировки валют, ценных бумаг и товарных активов. Отклонение фактического значения показателя от прогнозного может вызвать значительную волатильность на финансовых рынках.

Основные проблемы, с которыми сталкиваются пользователи экономических календарей:

- **Разрозненность источников данных.** Макроэкономические данные публикуются различными государственными ведомствами и организациями разных стран. Единого стандартизированного источника не существует.
- **Языковой барьер.** Описания событий зачастую доступны только на одном языке (чаще всего — английском или языке страны-публикатора), что затрудняет восприятие для международной аудитории.
- **Отсутствие гибкой фильтрации.** Многие существующие решения предоставляют ограниченные возможности по фильтрации событий по стране, дате, уровню важности.
- **Необходимость ручного обновления.** Ряд существующих календарей требует ручной актуализации данных, что приводит к задержкам и неточностям.

Постановка проблемы: необходимо разработать веб-приложение, которое автоматически агрегирует данные экономического календаря из нескольких внешних источников, предоставляет удобный мультиязычный интерфейс с системой фильтрации и обеспечивает периодическое обновление информации без участия пользователя.

### 1.2. Обзор существующих методов и решений

На рынке представлено несколько решений для отслеживания макроэкономических событий:

**Investing.com Economic Calendar** — один из наиболее популярных экономических календарей. Предоставляет широкий охват стран, фильтрацию по времени и важности, поддерживает множество языков. Недостатки: проприетарное решение, нет возможности локального развёртывания, реклама, зависимость от стороннего сервиса.

**ForexFactory Calendar** — ориентирован на трейдеров рынка Forex. Содержит подробные описания событий, историю значений. Недостатки: интерфейс на английском языке, ограниченная фильтрация по странам, отсутствие API для программной интеграции.

**TradingView Economic Calendar** — интегрирован в экосистему TradingView. Предоставляет визуально привлекательный интерфейс и ряд фильтров. Недостатки: доступен только в рамках платформы TradingView, невозможность независимого использования.

**FRED (Federal Reserve Economic Data)** — открытый API Федерального резервного банка Сент-Луиса. Предоставляет обширную базу экономических данных США с бесплатным API-доступом. Недостатки: охватывает только данные по США, требует программной обработки для представления в формате календаря.

**Alfa-Forex Экономический календарь** — русскоязычный экономический календарь с JSON API. Предоставляет данные по событиям множества стран с указанием волатильности, фактических и прогнозных значений, описаний на русском языке. Недостатки: привязка к одному источнику, отсутствие автономного клиентского приложения.

Сравнительный анализ существующих решений показывает, что ни одно из них не удовлетворяет всем выявленным требованиям одновременно. Проприетарные решения не позволяют осуществлять локальное развёртывание и модификацию. Открытые API (FRED) ограничены географически. Русскоязычные решения (Alfa-Forex) не предоставляют мультиязычного интерфейса.

### 1.3. Выбор подхода с обоснованием

На основании проведённого анализа принято решение о разработке собственного веб-приложения с клиент-серверной архитектурой по следующим причинам:

1. **Агрегация данных.** Приложение объединяет данные из нескольких источников (Alfa-Forex API, FRED API) в единую базу данных, обеспечивая полноту охвата.
2. **Открытость и расширяемость.** Архитектура системы спроектирована модульно, что позволяет добавлять новые источники данных без изменения существующего кода.
3. **Автономность.** Приложение может работать автономно, сохраняя данные в локальной базе и обновляя их по расписанию.
4. **Многоязычность.** Реализована поддержка четырёх языков интерфейса (русский, английский, китайский, испанский) с возможностью расширения.
5. **Современный стек технологий.** Использование FastAPI и React обеспечивает высокую производительность, масштабируемость и удобство разработки.

### Выводы по главе 1

В ходе анализа предметной области выявлена потребность в программном инструменте, агрегирующем макроэкономические данные из нескольких источников и предоставляющем единый мультиязычный интерфейс с гибкой фильтрацией. Обзор существующих решений показал, что ни одно из них не обеспечивает одновременно открытость, агрегацию данных, мультиязычность и возможность локального развёртывания. Обоснован выбор подхода на основе клиент-серверной архитектуры с использованием FastAPI и React.

## ГЛАВА 2. ПРЕДПРОЕКТНАЯ ЧАСТЬ

### 2.1. Диаграммы и моделирование

#### 2.1.1. Диаграмма вариантов использования (Use Case)

Система предусматривает взаимодействие с двумя категориями акторов: **Пользователь** (человек, работающий с интерфейсом) и **Внешний источник данных** (Alfa-Forex API, FRED API).

Основные сценарии использования:

**Пользователь:**
- UC-1: Просмотр списка экономических событий — пользователь открывает страницу «События» и видит табличное представление всех событий.
- UC-2: Фильтрация событий по стране — пользователь выбирает страну из выпадающего списка, отображаются только события выбранной страны.
- UC-3: Фильтрация событий по дате — пользователь выбирает временной диапазон (сегодня, завтра, неделя, точная дата).
- UC-4: Фильтрация событий по важности — пользователь выбирает уровень важности (низкая, средняя, высокая).
- UC-5: Просмотр детального описания события — пользователь нажимает на строку таблицы, открывается модальное окно с описанием.
- UC-6: Просмотр календарного представления — пользователь переходит на страницу «Календарь», где события сгруппированы по датам.
- UC-7: Смена языка интерфейса — пользователь выбирает язык из выпадающего списка (ru/en/zh/es).
- UC-8: Переключение темы оформления — пользователь переключает между светлой и тёмной темами.
- UC-9: Ручное добавление события — пользователь отправляет POST-запрос к API для создания нового события.

**Внешний источник данных:**
- UC-10: Автоматическая синхронизация данных — система периодически запрашивает данные из Alfa-Forex API и обновляет локальную базу.
- UC-11: Принудительное обновление — пользователь инициирует ручное обновление данных через API-эндпоинт `/events/refresh`.

#### 2.1.2. Функциональная модель IDEF0

Функциональная модель системы в нотации IDEF0 представлена на контекстном уровне (A-0) и уровне декомпозиции (A0).

**Контекстная диаграмма A-0:**
- Функция: «Управление экономическим календарём»
- Входы: запросы пользователя (HTTP-запросы), данные внешних источников (JSON-ответы Alfa-Forex API, FRED API)
- Выходы: отфильтрованные списки событий, детальные описания событий, статусы операций
- Механизмы: FastAPI-сервер, SQLite-база данных, React SPA-приложение, Playwright (для резервного скрапинга)
- Управление: конфигурация TTL синхронизации, правила фильтрации, валидация Pydantic-схем

**Декомпозиция A0 — основные функции:**
- A1: Приём и маршрутизация HTTP-запросов (FastAPI Router)
- A2: Синхронизация данных с внешними источниками (alfaforex_sync, fred_sync)
- A3: Операции CRUD с базой данных (crud.py, SQLAlchemy ORM)
- A4: Валидация и сериализация данных (Pydantic-схемы)
- A5: Отображение данных в пользовательском интерфейсе (React-компоненты)

#### 2.1.3. Диаграмма потоков данных (DFD)

**DFD уровня 0 (контекстная):**

Внешние сущности:
- Пользователь (браузер)
- Alfa-Forex API (внешний сервер)
- FRED API (внешний сервер)

Процесс: Система «Экономический календарь»

Хранилище данных: SQLite-база events.db

Потоки данных:
1. Пользователь → Система: HTTP GET /events (с параметрами фильтрации)
2. Система → Пользователь: JSON-массив событий
3. Alfa-Forex API → Система: JSON-массив событий (по HTTP)
4. FRED API → Система: JSON-массив релизов (по HTTP)
5. Система ↔ SQLite: чтение/запись записей таблицы events

**DFD уровня 1 (декомпозиция):**

Процессы:
- P1: Обработка запросов пользователя — получает HTTP-запросы, формирует ответы
- P2: Проверка актуальности данных — сравнивает метку последней синхронизации с TTL (по умолчанию 300 секунд)
- P3: Загрузка данных из Alfa-Forex — отправляет запрос к JSON API, парсит ответ, извлекает поля (IdEcoCalendar, Name, DateTime, Volatility, Country, Currency, DisplayActual, DisplayConsensus, DisplayPrevious, HTMLDescription)
- P4: Загрузка данных из FRED — запрашивает /fred/releases и /fred/releases/dates, формирует записи с автоматическим определением важности по ключевым словам
- P5: Upsert-операция — для каждого события проверяет наличие записи по external_id; если записи нет — вставляет, если есть и поля изменились — обновляет
- P6: Фильтрация и выборка — получает параметры фильтрации (country, regulator, importance), формирует SQL-запрос с условиями WHERE
- P7: Получение описания события — для не-русского языка запрашивает Alfa-Forex API с нужной культурой (en-US, zh-CN, es-ES), кэширует результат на 600 секунд

### 2.2. Выбор технологического стека

Система построена по архитектурному паттерну «клиент-сервер» с чётким разделением на backend (REST API) и frontend (SPA).

**Backend** представляет собой FastAPI-приложение, состоящее из следующих модулей:

- **main.py** — точка входа приложения, определяет маршруты API, настраивает CORS-middleware, инициализирует базу данных и запускает начальную синхронизацию при старте.
- **database.py** — модуль конфигурации базы данных: создание SQLAlchemy engine с подключением к SQLite, фабрика сессий SessionLocal, базовый класс Base для декларативных моделей, функция ensure_sqlite_columns() для лёгкой миграции схемы.

    importance: Mapped[str] = mapped_column(String(20), nullable=False, index=True)
    event_time: Mapped[Optional[str]] = mapped_column(String(16), nullable=True)
    remaining_time: Mapped[Optional[str]] = mapped_column(String(32), nullable=True)
    currency: Mapped[Optional[str]] = mapped_column(String(16), nullable=True)
    actual: Mapped[Optional[str]] = mapped_column(String(64), nullable=True)
    forecast: Mapped[Optional[str]] = mapped_column(String(64), nullable=True)
    previous: Mapped[Optional[str]] = mapped_column(String(64), nullable=True)
    description: Mapped[Optional[str]] = mapped_column(Text, nullable=True)
    source: Mapped[str] = mapped_column(

#### 3.2.3. API-маршруты

Основные эндпоинты FastAPI-приложения:

```python
@app.get("/events", response_model=list[EventRead])

    country: Optional[str] = Query(default=None),
    regulator: Optional[str] = Query(default=None),

Эндпоинт GET /events реализует отказоустойчивую логику: при включённом параметре auto_refresh данные автоматически обновляются из внешнего источника, если TTL истёк. ```python
def refresh_if_stale(db: Session, force: bool = False) -> Optional[dict[str, int]]:
    global _last_sync_epoch
    now = time.time()
    if not force and _last_sync_epoch and now - _last_sync_epoch < ALFAFOREX_TTL_SECONDS:
        return None
    result = sync_alfaforex_events(db)
    _last_sync_epoch = now
    return result
```

#### 3.2.5. Upsert-операция

Ключевой механизм обновления данных — upsert по полю external_id:

```python
def upsert_external_event(db: Session, *, external_id: str, title: str,
                           event_date, country: str, regulator: str,
                           importance: str, source: str, **kwargs) -> str:
    existing = db.scalar(
        select(models.Event).where(models.Event.external_id == external_id)
    )
    if existing is None:
        event = models.Event(
            title=title, date=event_date, country=country,
            external_id=external_id, source=source, ...
        )
        db.add(event)
        try:
            db.commit()
            return "inserted"
        except IntegrityError:
            db.rollback()
            return "skipped"

    changed = False
    for attr, value in (("title", title), ("date", event_date), ...):
        if getattr(existing, attr) != value:
            setattr(existing, attr, value)
            changed = True
    if not changed:
        return "skipped"
    db.add(existing)
    db.commit()
    return "updated"
```

Функция возвращает статус операции ("inserted", "updated", "skipped"), что позволяет агрегировать статистику синхронизации.

#### 3.3.4. Темы оформления

Поддерживаются две темы: тёмная (по умолчанию) и светлая. Тёмная тема использует тёмный фон (#070a16) с полупрозрачными поверхностями и градиентными акцентами фиолетового и бирюзового цветов. Светлая тема использует светлый фон (#f7f7fb) с карточками на белом фоне. Переключение тем реализовано через CSS-переменные и атрибут `data-theme` на элементе `<html>`. Выбор темы сохраняется в localStorage. При первом запуске учитывается системная настройка prefers-color-scheme.

#### 3.3.5. Типовой сценарий использования

1. Пользователь открывает приложение в браузере.
2. Загружается список экономических событий (при первом запуске выполняется синхронизация с Alfa-Forex).
3. Пользователь выбирает фильтр «Сегодня» для просмотра событий текущего дня.
4. Пользователь выбирает страну «США» для сужения выборки.
5. Пользователь нажимает на строку с событием «FOMC Interest Rate Decision».
6. Открывается модальное окно с описанием решения по процентной ставке.
7. Пользователь переключает язык на English для чтения описания на английском.
8. Система запрашивает описание на английском языке через API (culture=en-US).
9. Каждые 60 секунд данные обновляются автоматически.

### Выводы по главе 3

Реализовано полнофункциональное веб-приложение, соответствующее сформулированным требованиям. Backend обеспечивает REST API с пятью эндпоинтами, отказоустойчивую синхронизацию данных с двухуровневой стратегией (JSON API + Playwright-скрапинг), upsert-логику по external_id и TTL-кэширование. Frontend предоставляет адаптивный интерфейс с фильтрацией, двумя представлениями данных (таблица и календарь), четырёхъязычной интернационализацией и двумя цветовыми темами. Система соответствует всем 13 функциональным и 7 нефункциональным требованиям, определённым в главе 2.

В ходе выполнения курсовой работы была достигнута поставленная цель — спроектировано и реализовано полнофункциональное веб-приложение экономического макро-календаря с автоматической синхронизацией данных из внешних источников.

Были решены все поставленные задачи:

1. **Проведён анализ предметной области** — выявлена потребность в инструменте агрегации макроэкономических данных из нескольких источников с мультиязычным интерфейсом и гибкой фильтрацией.

2. **Определены требования** — сформулированы 13 функциональных и 7 нефункциональных требований к системе.

3. **Спроектирована архитектура** — разработана клиент-серверная архитектура с чётким разделением на backend (FastAPI REST API) и frontend (React SPA), построены диаграммы Use Case, IDEF0, DFD.

4. **Реализована серверная часть** — создан FastAPI-сервер с 5 эндпоинтами, ORM-моделью на SQLAlchemy 2.0, Pydantic-валидацией, модулями синхронизации для Alfa-Forex (с резервным Playwright-скрапингом) и FRED API.

5. **Реализована клиентская часть** — создано React-приложение с двумя представлениями данных (таблица с фильтрами и календарь), модальными окнами описаний, четырёхъязычной интернационализацией (ru/en/zh/es), двумя цветовыми темами и автоматическим обновлением данных.

6. **Настроен CI/CD** — реализовано автоматическое развёртывание фронтенда на GitHub Pages через GitHub Actions.

Ключевые технические решения, обеспечивающие надёжность системы:
- Двухуровневая стратегия синхронизации (JSON API + Playwright-скрапинг).
- TTL-механизм с настраиваемым интервалом обновления (по умолчанию 300 секунд).
- Upsert-логика по external_id, предотвращающая дублирование данных.
- Отказоустойчивая обработка ошибок внешних источников (API остаётся доступным при сбоях синхронизации).
- Кэширование переводов описаний на 600 секунд.

**Возможные пути дальнейшего развития и совершенствования модуля:**

1. **Интеграция FRED API в runtime** — подключение модуля fred_sync.py к маршрутам API для расширения охвата экономических данных США.
2. **Добавление WebSocket-уведомлений** — оповещение пользователя в реальном времени при появлении новых событий или обновлении значений.
3. **Реализация полнотекстового поиска** — добавление возможности поиска событий по названию и описанию.
4. **Расширение языковой поддержки** — добавление немецкого, французского, арабского и других языков.
5. **Добавление авторизации** — разграничение доступа для администраторов (управление событиями) и обычных пользователей (просмотр).
6. **Миграция на PostgreSQL** — переход на более производительную СУБД для масштабирования при росте нагрузки.
7. **Реализация push-уведомлений** — отправка уведомлений о предстоящих важных событиях через Service Worker.
8. **Добавление графиков исторических значений** — визуализация динамики макроиндикаторов.
